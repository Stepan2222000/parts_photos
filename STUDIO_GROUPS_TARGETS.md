# Studio: группы, item-привязка, перенос результатов

Спека по тому, как Studio после генерации предлагает перенос результатов
в правильные коллажи внутри правильных групп. Описывает обсуждённую
архитектуру и решения. По этому тексту можно реализовывать.

## Контекст

Сейчас Studio:
- Принимает фото (загрузка / выбор из коллажа)
- Гонит их через codex (`gpt-image-2`), получает апгрейднутую версию
- Ищет среди существующих коллажей те, у которых `owner_id` или артикул
  совпадает с именем файла → показывает баннер «Совпадения по статьям»
- Перенос делает копированием/перемещением файла из studio bucket в photos
  bucket + создание новой строки `photos`

Что не работает / не покрыто:
- Если **подходящего коллажа ещё нет** — пользователю надо идти в Photos,
  заводить руками, возвращаться и переносить. Долго.
- Группы у нас имеют **разный смысл** (Эталонные, Реальные, Дефектные,
  Avito, Поступления), и Studio не понимает в какую можно лить, а в
  какую — нет.
- Реальные/дефектные группы по сути привязаны не к smart-парту, а к
  **конкретному физическому экземпляру** (item в `parts_uchet`).
- Нет связи parts_photos ↔ parts_uchet.

## Группы — роли и параметры

Группы в `photo_groups` сейчас единообразные. По смыслу делятся на
несколько ролей. Эти роли **не лежат в БД** — это **конфиг в коде**
(словарь `group_id → settings`). Меняется только при разворачивании
новых групп. Менять налету через UI не нужно.

### Текущий расклад

| Группа | studio_role | owner_kind | defect_filter | is_reference |
|--------|-------------|------------|---------------|--------------|
| Эталонные на публикацию | source | smart_part | any | true |
| Реальные фотографии | source | instance | without | false |
| Дефектные фотографии | source | instance | with | false |
| Реальные на публикацию | **target** | instance | without | false |
| Дефектные на публикацию | **target** | instance | with | false |
| Avito 2-й | **target** | instance | any | false |
| Поступления | вне Studio | tracking (внешнее) | — | false |

### Что значат поля

- **`studio_role`** — участвует ли группа в Studio пайплайне:
  - `source` — из коллажей этой группы можно брать фото в Studio как
    исходники (через таб «Из коллажа» или через quick-action на фото)
  - `target` — в коллажи этой группы можно переносить результаты Studio
  - `none` — группа Studio не касается

- **`owner_kind`** — что является «владельцем» коллажа:
  - `smart_part` — коллаж агрегирует все фото запчасти на уровне
    `smart.parts.id` (в одной группе на один smart-парт = один коллаж)
  - `instance` — коллаж привязан к конкретному физическому экземпляру
    `parts_uchet.items.id` (в одной группе на один item = один коллаж)
  - `tracking` — связь с `delivery.parcels_mutations.tracking_number`,
    Studio это не использует, упомянуто только для полноты
  - `box` — внутренний концепт `parts_uchet` (коробка), Studio не
    использует

- **`defect_filter`** — для item-уровневых групп фильтрует доступные
  экземпляры по `parts_uchet.items.defect`:
  - `without` — только `defect=false` (например «Реальные на публикацию»)
  - `with` — только `defect=true` (например «Дефектные на публикацию»)
  - `any` — без фильтра (например «Avito 2-й» — туда что угодно)

- **`is_reference`** — существующий флаг: фото в коллажах нельзя
  редактировать, это эталонные данные из smart. Перпендикулярно
  `studio_role` — Эталонные могут быть source для Studio, но сами
  read-only.

### Поведение при изменении настроек группы

Настройки в коде. Если меняется `defect_filter` или `owner_kind` для
существующей группы — **уже существующие коллажи не валидируем и не
трогаем**. Новая логика применяется только при поиске/создании
коллажей под новые джобы.

## Item ≠ smart_part

`parts_uchet.items` — таблица физических экземпляров:
- `id` (integer, например 412) — натуральный ID экземпляра
- `smart_part_id` (text, формат `smart_NNNNNNNN`) — ссылка на запчасть
  в smart
- `defect` (bool), `defect_note` (text), `note` (text)
- `box_id`, `status`, `is_in_transit` и пр. — для нашего пайплайна не
  важно, в Studio не показываем

Один smart_part может иметь **несколько items**.

### Связь parts_photos ↔ parts_uchet

Сейчас её нет. Сделаем тем же паттерном, что уже есть для smart:
**postgres_fdw**, схема `uchet_ext`, foreign table `uchet_ext.items`
с минимальным набором полей (`id`, `smart_part_id`, `defect`,
`defect_note`, `note`).

`box_id` и метаданные коробки тянуть не надо — это внутренний учёт.

## Article matching: filename → item-кандидаты

После генерации worker запускает поиск кандидатов на перенос. Логика:

1. Берём `studio_jobs.source_filename` (если есть). Нормализуем —
   lowercase, убираем всё что не [a-z0-9]. Если короче 4 символов —
   пропускаем (как сейчас).
2. Ищем в `smart_ext.parts` запчасть, у которой нормализованный
   `id` или один из `articles[]` совпадает с filename. Если нет —
   suggestions пустые. **Без артикула в smart никаких предложений
   быть не может.**
3. Нашли smart_part_id. Достаём из `uchet_ext.items` все экземпляры с
   этим `smart_part_id`.
4. Для каждой target-группы (`studio_role='target'`) применяем
   `defect_filter` — оставляем подходящие items.
5. Для каждого подходящего item проверяем — есть ли уже коллаж в этой
   группе (`photo_collages WHERE owner_kind='instance' AND
   owner_id=item.id::text AND group_id=$g`). Если есть — это **target
   существующий**, если нет — это **target создадим**.
6. Складываем всё в `studio_jobs.suggested_collages_json` в виде:
   ```json
   {
     "smart_part_id": "smart_10000090",
     "smart_part_name": "Водяной насос",
     "matched_article": "ABC-123",
     "items_by_group": {
       "<group_id A>": [
         {"item_id": 412, "defect": false, "defect_note": null,
          "existing_collage_id": "uuid|null"}
       ],
       "<group_id B>": [
         {"item_id": 412, "defect": false, "defect_note": null,
          "existing_collage_id": null}
       ]
     }
   }
   ```

## Owner_id для item-коллажа

`item.id` — целое число, `photo_collages.owner_id` — text. Пишем
**голое число как строку**: `"412"`. Без префиксов вроде `item_412`
или `smart_10000090#412`. `owner_kind='instance'` уже отделяет от
smart_part'ов.

UI показывает «Item #412» через явный formatter, не через сам
owner_id.

## Имя коллажа в UI (для item-коллажа)

Главная строка: **`Item #412`**

Подпись (мелким): **`smart_10000090 · Водяной насос`** (smart_part_id +
`smart.parts.name`)

Если `defect=true` — рядом маркер дефекта + первые ~50 символов
`defect_note`. Без `box_id` — он не нужен в UI.

## Создание батча — что меняется

Поле «Куда положить результаты после генерации» (target_collage_id)
из формы создания батча **убирается**. Группа выбирается **только
после генерации**, на экране результатов. В БД поле
`studio_batches.target_collage_id` тоже не нужно — удалить столбец.

При создании батча пользователь только:
- грузит файлы / выбирает из коллажа
- ставит галки опций + custom_prompt
- (если нужно) выбирает фон и вотермарк

Нажимает Generate → batch уходит в очередь.

## Экран результатов — фильтр по target-группам

После генерации (или сразу когда открываешь батч) экран показывает
все джобы как карточки. Сверху **табы / dropdown по target-группам**:
для каждой группы со `studio_role='target'`. Без дефолтного
выделения — пользователь сам выбирает с какой группы начать.

Когда выбрана группа, для каждой карточки джоба внизу появляется
строка-предложение одного из видов:

- ✅ **«→ Item #412»** — коллаж в этой группе уже есть. Чекбокс
  поставлен по умолчанию.
- ➕ **«создать → Item #412»** — item подходит по `defect_filter`
  группы, коллажа нет. Будет создан. Чекбокс поставлен по умолчанию.
- ❓ **«3 экземпляра — выбери ▾»** — у smart_part несколько
  подходящих items. Dropdown внутри карточки. По умолчанию первый
  выбран. Чекбокс поставлен.
- ❌ **«нет дефектных»** / **«артикул не в smart»** /
  **«имя файла не распознано»** — серое, чекбокс снят. Кликабельно
  для ручного выбора (см. ниже).

Внизу экрана **кнопка `Перенести выбранные (N)`**. По нажатию:
1. Сервер создаёт недостающие коллажи (`existing_collage_id=null` →
   INSERT в `photo_collages` с `owner_kind='instance'`,
   `owner_id=item.id::text`, в текущей группе).
2. Для каждого джоба переносит файл из studio bucket в photos bucket
   под ключ `groups/{group_id}/collages/{collage_id}/{photo_id}.png`,
   создаёт `photos` строку, проставляет
   `studio_jobs.transferred_to_photo_id`.

Один джоб = один перенос. Если хочешь то же фото в двух группах —
переключись на другую target-группу и сделай ещё один перенос (но
текущая модель `transferred_to_photo_id` это **запретит**, см.
следующий блок).

## Один джоб → один перенос

`studio_jobs.transferred_to_photo_id` остаётся single-reference. После
переноса джоб становится «использованным» — кнопка переноса по нему
больше не показывается, на карточке метка «✓ Перенесено в [группа]».

Если в будущем захочется переносить одно и то же фото в несколько
групп (например в Реальные на публикацию И в Avito 2-й одновременно)
— **тогда** заведём `studio_job_transfers (job_id, photo_id, group_id,
transferred_at)`. Пока — нет.

## Ручной выбор когда «не распознано»

Клик по серой строке-предложению открывает поиск **запчасти** (по
артикулу или smart-id). Тот же `CollageInlineSearch`, что уже
используется. Пользователь выбрал запчасть → backend делает то же,
что в шагах 3-5 article matching: достаёт items, фильтрует по
defect_filter группы, подставляет первый/раскрывает выбор. Строка
становится зелёной, чекбокс ставится. После «Перенести выбранные»
обрабатывается как обычное предложение.

## Кнопка «Апгрейднуть»

В Photos UI на каждом фото в коллаже **source-группы**
(`studio_role='source'`) показывается иконка «Апгрейднуть в Studio»
(уже есть как Sparkles в `PhotosGrid.tsx`). Клик ведёт на `/studio`
с пред-выбранным этим фото как source.

Кнопки «Апгрейднуть весь коллаж» **пока нет** — только per-photo.

## Удаление и каскады

При **удалении батча**:
- БД: каскадно удаляются `studio_jobs` (FK CASCADE).
- БД: `photos.studio_job_id → studio_jobs ON DELETE SET NULL` →
  перенесённые в коллажи фото остаются (теряют только обратную ссылку).
- MinIO: файлы в `parts-photos-studio/uploads/{batch_id}/` и
  `results/{batch_id}/` **сейчас не чистятся** (известный orphan-долг,
  отдельная задача).

При **удалении item в parts_uchet** (если когда-нибудь будет такое
действие): instance-коллажи в parts_photos осиротеют. Нужна периодика
сверки или soft-delete на стороне parts_uchet. Не делаем сейчас.

## Что НЕ делаем

- m2m связь джоб ↔ photos (один джоб → один перенос)
- UI редактирования настроек групп (роли в коде)
- Заведение items из Studio (только в parts_uchet)
- Поддержка Поступлений в Studio (track-фото — отдельный пайплайн)
- Пометка `items.defect` из Studio (не редактируем учёт)
- Кнопка «Апгрейднуть весь коллаж»
- Дефолтный таргет на экране результатов (показываем без выделения)
- Хранение настроек групп в БД (только в коде)

## Что нужно сделать (план реализации)

В порядке логических зависимостей:

1. **FDW в parts_uchet.** Аналог `fdw_smart.sql`: новая схема
   `uchet_ext`, foreign table `uchet_ext.items` с минимальными
   полями. SQL-миграция или shell-скрипт.

2. **Конфиг групп в коде.** В backend — словарь
   `GROUP_SETTINGS: dict[UUID, GroupConfig]`, либо читаем из конфиг-
   файла. Хранит studio_role / owner_kind / defect_filter для
   каждого known group_id. Helper-функции:
   `studio_targets() → list[group_id]`, `is_studio_source(group_id)`,
   `defect_filter(group_id)` и т.п.

3. **Расширение CHECK constraint** на `photo_collages.owner_kind`:
   добавить `'box'` и `'tracking'` в список (на будущее, чтобы при
   ручных вставках принималось). Можно отложить, если не нужно.

4. **Новая логика matching.** `find_matches` в `article_match_db.py`
   возвращает не плоский список коллажей, а структуру по группам и
   items. Worker сохраняет в `suggested_collages_json` в новом формате.

5. **API.** Расширить:
   - `GET /studio/batches/{id}` отдаёт по каждому джобу новую
     структуру suggestions
   - `POST /studio/batches/{id}/transfer` (заменяет
     `transfer-suggested`) — принимает `[{job_id, group_id, item_id}]`.
     Если item уже имеет коллаж в группе — переносит туда. Если нет —
     создаёт коллаж и переносит. Делается транзакционно (как есть).
   - `GET /collages/search` (для ручного выбора запчасти) — должен
     уметь искать по smart-id и по артикулу. Уже умеет, проверить.
   - Новый эндпоинт типа `GET /studio/lookup-by-part?smart_part_id=X
     &group_id=Y` — возвращает items с применённым defect_filter и
     инфой о существующих коллажах.

6. **DB-миграция.** Удалить `studio_batches.target_collage_id`
   (поле больше не используется).

7. **Frontend.**
   - Убрать «Куда положить результаты» из формы создания батча
     (`SourcesPanel.tsx`).
   - Заменить `TransferSuggestions` на новый компонент с табами
     target-групп. Каждая карточка джоба получает строку-
     предложение зависимости от выбранной группы. Per-card
     dropdown для выбора item, ручной search для нераспознанных.
   - Расширить `BatchView.tsx` чтобы передавать group selection
     в новый компонент и в `transfer` API.
   - В `PhotosGrid.tsx` оставить existing «Апгрейднуть» иконку, она
     уже есть.
