# Спека: привязка коллажей к конкретным item'ам учёта (instance owners)

Статус: **черновик, согласован устно, к реализации.**
Принцип проекта, действующий и здесь: **никаких тихих/глупых фолбеков** — любая
неоднозначность или несоответствие завершается явной ошибкой с понятным
сообщением, а не «подставим что-нибудь по умолчанию».

---

## 1. Контекст и проблема

В фотобанке у каждого коллажа (`photo_collages`) есть владелец, заданный парой
`owner_kind` + `owner_id`. Владелец бывает двух видов:

- **`smart_part`** — владелец это позиция каталога `smart` (`smart_ext.parts`,
  id вида `smart_10001062`). Коллаж означает «фотографии этого артикула вообще,
  на уровне номенклатуры»: одна и та же позиция, независимо от того, сколько
  физических штук пришло на склад. Имя и артикулы подтягиваются из smart через
  postgres_fdw.
- **`instance`** — владелец это **конкретная физическая единица товара** из
  системы учёта (`uchet_ext.items`, целочисленный `id`). У такой единицы есть
  собственное состояние: флаг `defect`, заметка `defect_note`, складской
  `status`, произвольная `note`, и ссылка `smart_part_id` на номенклатуру.
  Коллаж означает «фотографии именно этой штуки», с её реальным состоянием.

**Проблема, которую закрываем.** Сейчас ручное создание коллажа («+ New
collage» → `CreateCollageDialog`) **всегда** отправляет `owner_kind:
"smart_part"` и ищет владельца только по smart-каталогу, **игнорируя
назначение группы**. Бэкенд `create_collage` тоже не сверяет вид владельца с
группой — пишет что прислали. Из-за этого в группах, которые по смыслу должны
держать конкретные физические экземпляры (Реальные/Дефектные фотографии и
…на публикацию), оказываются smart-парт коллажи. Это не соответствует задумке:
реальное или дефектное фото показывает **конкретную единицу** с её состоянием,
а не абстрактный номер.

**Цель.** При создании коллажа в instance-группах владелец должен выбираться
как **конкретный item учёта**, с поиском двумя путями: (а) нашли smart-парт по
номеру/названию/артикулу → затем выбираем нужный экземпляр; (б) сразу ищем по
id экземпляра.

## 2. Доменная модель

### 2.1 `smart_ext.parts` (каталог smart, read-only через FDW)

| Колонка | Тип | Назначение |
|---|---|---|
| `id` | text | `smart_NNNNNNNN` |
| `name` | text | человекочитаемое название |
| `articles` | text[] | список артикулов |
| `is_draft` | bool | черновик (не фильтруем) |

### 2.2 `uchet_ext.items` (учёт, read-only через FDW)

| Колонка | Тип | Назначение |
|---|---|---|
| `id` | integer | уникальный id физического экземпляра |
| `smart_part_id` | text | к какой позиции smart относится |
| `defect` | bool | бракованный/дефектный экземпляр |
| `defect_note` | text \| null | описание дефекта |
| `status` | text | складской статус (`in_stock`, …) |
| `note` | text \| null | произвольная заметка |

Важно: у item **нет собственного имени**. В UI его представляем как
`#<id> · <название smart-парта> · <артикул>` плюс бейдж дефекта и заметка.
Название и артикул берём джойном к `smart_ext.parts` по `smart_part_id`.

### 2.3 Хранение владельца

`photo_collages.owner_id` — это `text`. Для `smart_part` там `smart_NNN`, для
`instance` там **строковое представление целого id экземпляра** (`"480231"`).
Так уже хранят коллажи, создаваемые Studio-переносом (`article_match_db.py`
сравнивает `owner_id = ANY($items_text)`), так что формат совместим.

## 3. Конфигурация групп (единственный источник истины)

Роли и вид владельца групп заданы как **config-as-code** в
`api/app/studio/groups.py` → `GROUP_SETTINGS`. Это карта `group_uuid →
(studio_role, owner_kind, defect_filter, accepts_defects)`. Новая группа или
смена роли — это правка кода/PR, не запрос в API. Никаких параллельных
списков групп заводить нельзя — иначе расхождения.

| Группа | `studio_role` | **`owner_kind`** | `defect_filter` |
|---|---|---|---|
| Эталонные на публикацию | target | **smart_part** | any (дефекты запрещены) |
| Avito 2-й аккаунт | target | **smart_part** | any |
| Реальные на публикацию | target | **instance** | without |
| Дефектные на публикацию | target | **instance** | with |
| Реальные фотографии | source | **instance** | without |
| Дефектные фотографии | source | **instance** | with |
| Поступления | none (вне Studio) | instance | any |

Item-пикер включается **ровно там, где `owner_kind == "instance"`**: это
Реальные/Дефектные фотографии и Реальные/Дефектные на публикацию. Эталонные и
Avito остаются на smart-парт пикере без изменений.

`defect_filter` — жёсткое правило отбора экземпляров для группы:
- `with` → только `item.defect = true`;
- `without` → только `item.defect = false`;
- `any` → ограничения по дефекту нет (актуально лишь для smart-групп).

## 4. Поведение создания коллажа

### 4.1 Выбор сценария по группе

`CreateCollageDialog` (и бэкенд) узнают вид владельца из конфига группы:

- **smart-группа** (`owner_kind = smart_part`): текущее поведение без
  изменений — поиск по smart-каталогу, владелец = `smart_NNN`.
- **instance-группа** (`owner_kind = instance`): новый item-пикер (раздел 5),
  владелец = id экземпляра.

Фронт получает `owner_kind` и `defect_filter` группы из расширенного ответа
`GET /groups` (раздел 7.1). `/studio/target-groups` для этого **не подходит** —
там только target'ы, а source-группы (Реальные/Дефектные фотографии) в нём
отсутствуют, хотя в них тоже создаём instance-коллажи.

### 4.2 Жёсткое навязывание вида владельца (фикс бага)

`create_collage` обязан сверять присланный `owner_kind` с конфигом группы:

- если группа есть в `GROUP_SETTINGS` и `payload.owner_kind !=
  cfg.owner_kind` → **422** с пояснением (никакого «ну ладно, запишем как
  есть»);
- если группы **нет** в `GROUP_SETTINGS` → это ошибка конфигурации, а не повод
  угадывать → **422** «group is not configured for collage creation». Не
  откатываемся молча на smart_part.

Это и закрывает исходный баг: smart-парт коллаж в instance-группу больше не
проходит.

## 5. UX пикера владельца в instance-группе

Одно поле ввода с **авто-режимом** по содержимому запроса:

1. **Запрос состоит только из цифр** → трактуем как **поиск по id
   экземпляра**. Резолвим item, показываем карточку: `#<id>`, название и
   артикул его smart-парта, бейдж дефекта, `note`, и метка «уже есть коллаж»
   если в этой группе он уже заведён. Выбор — сразу этот экземпляр.
2. **Запрос содержит буквы** → **поиск smart-парта** по `smart_id` / названию /
   артикулу (как сейчас). Список парт-кандидатов. При выборе парта раскрываем
   **второй шаг — список его экземпляров**, уже отфильтрованных по
   `defect_filter` группы, каждый со своим `#id`, бейджем дефекта, заметкой и
   меткой «уже есть коллаж». Выбираешь конкретный экземпляр.

Правила отображения и блокировок (без глупых фолбеков):

- Экземпляр, **не проходящий `defect_filter`** группы (например, дефектный в
  «Реальные …»), показывается, но **выбор заблокирован** с явной подписью
  «дефект не подходит для этой группы». Не прячем молча и не «подменяем» на
  допустимый.
- Экземпляр, у которого **уже есть коллаж** в этой группе, помечается и при
  попытке выбрать ведёт на существующий коллаж (а не создаёт дубль).
- Пустой результат поиска — явное «ничего не найдено», без авто-создания
  чего-либо.

Источник истины по дефект-фильтру и «existing collage» — серверные lookup-
эндпоинты (раздел 7), фронт сам ничего не доугадывает.

## 6. Правила валидации на бэкенде (без глупых фолбеков)

`validate_owner_exists(kind, owner_id, group_id)` расширяется:

- **smart_part**: как сейчас — `owner_id` обязан существовать в
  `smart_ext.parts`, иначе 422.
- **instance**:
  1. `owner_id` обязан парситься в целое — иначе 422 «item_id must be integer».
  2. Экземпляр обязан существовать в `uchet_ext.items` — иначе 422 «item not
     found».
  3. Экземпляр обязан проходить `defect_filter` группы — иначе 422 «item defect
     state does not match group».
  4. Экземпляр обязан иметь `status = 'in_stock'` — иначе 422 «item is not in
     stock». (Согласовано: фильтруем по `in_stock` во всех instance-группах,
     включая «…на публикацию». Если по ходу выяснится, что публикуемым нужны и
     не-in_stock экземпляры — ослабим явным изменением спеки, а не тихим
     обходом.)

Все ошибки — явные HTTP-коды с текстом. Никаких «не нашли — создадим
заглушку», «не подходит — притворимся что подходит».

## 7. Контракты эндпоинтов

### 7.1 `GET /groups` (расширяется)

К каждому элементу добавляются поля из `GROUP_SETTINGS`:

```
owner_kind:    "smart_part" | "instance"
defect_filter: "with" | "without" | "any"
```

Берутся из конфига по `group.id`. Если группы нет в конфиге — поля
отсутствуют/`null`, и фронт трактует это как «создание коллажей не
сконфигурировано» (кнопку создания можно дизейблить).

### 7.2 `GET /studio/lookup/items?smart_part_id=&group_id=` (уже есть)

Шаг 2 для текстового поиска: отдаёт экземпляры smart-парта, прошедшие
`defect_filter` группы, с `existing_collage_id`. Переиспользуем как есть.
Контракт элемента: `{ item_id, defect, defect_note, existing_collage_id }`.

### 7.3 `GET /studio/lookup/item?item_id=&group_id=` (**новый**)

Прямой поиск по id экземпляра. Возвращает:

```
{
  item_id: int,
  smart_part_id: str,
  smart_part_name: str | null,
  smart_part_articles: str[],
  defect: bool,
  defect_note: str | null,
  status: str,
  passes_filter: bool,        // проходит ли defect_filter этой группы
  in_stock: bool,
  existing_collage_id: uuid | null
}
```

Если item не существует — 404 (явно). `passes_filter`/`in_stock` отдаём
честно, чтобы фронт мог заблокировать выбор с понятной причиной, а не
угадывать.

### 7.4 `POST /collages` (правила ужесточаются)

Тело без изменений (`group_id, owner_kind, owner_id`), но добавляются проверки
из разделов 4.2 и 6. Уникальность — существующая `(group_id, owner_kind,
owner_id)`.

---

## 8. Уникальность и пограничные случаи

- **Один экземпляр — один коллаж в пределах группы.** Гарантируется
  `UNIQUE(group_id, owner_kind, owner_id)`. В разных instance-группах один и
  тот же item даёт разные коллажи (это норм: «реальное фото» и «дефектное
  фото» — разные наборы).
- **Item сменил дефект-статус после создания коллажа.** Коллаж не
  перепривязывается автоматически. Это осознанное решение: фотографии
  принадлежат экземпляру, а его текущее состояние — отдельный вопрос. При
  необходимости синхронизации заведём отдельную задачу.
- **Группа отсутствует в `GROUP_SETTINGS`.** Создание коллажа запрещено (422),
  а не «по умолчанию smart_part».
- **Прямой поиск по id даёт экземпляр чужого дефект-класса.** Показываем, выбор
  блокируем (см. раздел 5), коллаж не создаём.

---

## 9. Что НЕ входит в эту итерацию

- Чистка «неправильных» smart-парт коллажей в instance-группах — на живом
  сервере их нет, после фикса не появятся; отдельная задача при необходимости.
- Перепривязка коллажей при смене дефект-статуса экземпляра.
- Изменение Studio-переноса — он уже создаёт instance-коллажи корректно.
- Поиск экземпляров по `note`/свободному тексту (пока только smart-поиск и
  прямой id).

## 11. Верификация

1. **smart-группы не сломаны:** создать коллаж в «Эталонные»/«Avito» — поиск по
   smart, владелец smart_part, как раньше.
2. **instance-группа, текстовый поиск:** в «Реальные на публикацию» ввести
   название/артикул → выбрать парт → увидеть только бездефектные экземпляры →
   выбрать item → коллаж создан с `owner_kind=instance`, `owner_id=<item id>`.
3. **instance-группа, поиск по id:** ввести цифровой id → карточка экземпляра →
   создать коллаж. Ввести id дефектного экземпляра в «Реальные …» → выбор
   заблокирован с явной подписью.
4. **Навязывание owner_kind:** POST `/collages` с `owner_kind=smart_part` в
   instance-группу → 422 (а не запись).
5. **Несуществующий item:** POST с `owner_id` несуществующего id → 422.
6. **Дубль:** повторное создание для того же (группа, item) → 409, либо UI
   ведёт на существующий коллаж.
