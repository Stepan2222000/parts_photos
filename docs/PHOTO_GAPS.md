# Пробелы фото («Gaps»)

Страница для заполнения **пробелов** в публикационных каналах по тому, что
реально лежит на складе (`parts_uchet`, `uchet_ext.items`, `status='in_stock'`).
Цель — переиспользовать готовую логику (матчинг, перенос, Studio-апгрейд) и
закрывать места, где фото не хватает, по принципу DRY.

## 1. Что такое пробел

Пробел = **ноль живых фото** (`photos.state='uploaded'`) в целевом канале для
того, что есть в наличии. Три категории (вкладки):

| Вкладка | Канал-цель | Владелец | Источник пробелов (in_stock) |
|---------|-----------|----------|-------------------------------|
| **Эталонные** | Эталонные на публикацию (`ae69…`) | smart_part | smart_part'ы, у которых есть ≥1 экземпляр `condition='new'` и нет эталонных фото |
| **Personal** | На публикацию (`3cf6…`) | instance | экземпляры `condition='personal'` без фото на публикацию |
| **Defect** | На публикацию (`3cf6…`) | instance | экземпляры `condition='defect'` без фото на публикацию |

«Ноль живых фото» = нет коллажа ИЛИ коллаж есть, но в нём ноль `uploaded`-фото.
Существующий пустой целевой коллаж переиспользуется (find-or-create под
advisory-lock), не плодим дубли.

Avito 2-й аккаунт (тоже smart_part-витрина) сейчас НЕ показывается, но вся
бэкенд-логика обобщена по target-группе — добавляется одной строкой конфига.

## 2. Источники для заполнения

| Вкладка | Авто-источники | Ручной escape-hatch |
|---------|----------------|----------------------|
| Эталонные (smart_part `S`) | реальные фото всех `in_stock` **non-defect** экземпляров `S` + Свободные коллажи, привязанные к `S` | поиск/выбор любых фото |
| Personal/Defect (item `I`) | реальные фото **этого же** экземпляра `I` + Свободные коллажи его smart_part | поиск/выбор **любых** фото (через силу) |

defect-сырьё не идёт в Эталонные — гейт `condition_allowed` + per-item
defect-guard (как в Studio-переносе). Дефектный источник в эталон → 400.

## 3. Два действия и судьба источника

Действий ровно два, они разные по природе:

- **Перенос** — физическое размещение готового фото (move/copy, без ИИ).
- **Апгрейд** — Studio генерирует **новое** ИИ-изображение из исходного; приёмка
  результата — штатным шагом Studio (`POST /studio/batches/{id}/transfers`).

Судьбу исходника решает пара **(канал-источник → канал-цель)**, а не способ.
Единый инвариант (без отдельной таблицы — переиспользуем `is_direct_move_allowed`):

> **Источник гасится ⟺ пара является direct-move маршрутом
> (`DIRECT_MOVE_TARGETS`), т.е. ровно Реальные → На публикацию.**

| Откуда → Куда | Перенос | Исходник после переноса | Апгрейд | Исходник после приёмки |
|---------------|---------|--------------------------|---------|------------------------|
| Реальные → На публикацию | move | **уходит** | да | **уходит** (гасится при приёмке) |
| Реальные → Эталонные | copy | **остаётся** | да | **остаётся** |
| Свободные → Эталонные | copy | **остаётся** | да | **остаётся** |
| Свободные → На публикацию | copy | **остаётся** | да | **остаётся** |

- **move**: тот же `photos`-ряд репойнтится в целевой коллаж, объект в MinIO
  переезжает, в источнике ничего не остаётся.
- **copy**: объект MinIO дублируется, создаётся новый `photos`-ряд
  (`source='copy'`, `previous_s3_key` = исходный ключ), оригинал цел.
- **апгрейд-гашение**: после приёмки ИИ-результата на витрину, если пара —
  direct-move (Реальные → На публикацию), исходное сырьё soft-удаляется
  (`state='deleted'`) и объект MinIO стирается. На неудачной/непринятой
  генерации сырьё не трогаем — потерь нет.

Один сырой кадр в две витрины: move/гашение отдаёт его в первую; для второй
сырья уже нет — осознанное жёсткое ограничение.

## 4. API

Все маршруты на FastAPI-приложении (`api/app`).

- `GET /gaps/counts` → `{reference, personal, defect}` — счётчики вкладок.
- `GET /gaps?kind=reference|personal|defect&q=&limit=&offset=` → список строк
  пробелов с полями идентичности (`smart_part_id`/`item_id`, `name`,
  `articles`, `condition`), счётчиками доступных источников (`real_photos`,
  `free_collages`) и `target_collage_id` существующего пустого коллажа (или null).
- `GET /gaps/sources?kind=…&smart_part_id=…|item_id=…` → `{real:[…], free:[…]}`
  списки коллажей-источников с фото для пикера.
- `POST /gaps/fill` `{target_group_id, target_owner_kind, target_owner_id,
  photo_ids[]}` → размещает фото. Per-photo op (move/copy) выводится сервером из
  `is_direct_move_allowed(src_group, target_group)`. Валидации без фолбеков:
  owner_kind совпадает с каналом, item `in_stock` + `condition` под фильтр
  канала, маршрут разрешён `is_transfer_allowed`, defect-guard для эталонных,
  видео на публикацию нельзя.
- Апгрейд переиспользует существующее: `POST /studio/batches`
  (`source_photo_ids` → батч), приёмка `POST /studio/batches/{id}/transfers`.

### Переиспользование (DRY)

- Матчинг — `studio.article_match_db.find_matches` / `_build_by_group` /
  `search_items` (вкл. `free=True` для библиотеки).
- Перенос — общий core `_place_photos` (collages.py): и `POST /gaps/fill`, и
  существующий `POST /collages/{id}/transfer` зовут его (transfer = частный
  случай move Реальные → На публикацию того же экземпляра).
- Гашение при апгрейде — в `studio._do_transfer` после размещения результата,
  условие `is_direct_move_allowed(source_group_id, target_group_id)`.

## 5. Схема БД

Миграция `migration_009_photos_source_copy.sql`: расширить
`photos_source_check` до `IN ('upload','studio','copy')`. Других изменений схемы
нет — пробелы это чистые запросы, перенос/копия живут в существующих таблицах.

## 6. UI

Страница `/gaps` (TOOLS в сайдбаре, рядом с «Search photos»/«Studio»). Сверху
сегмент-переключатель трёх вкладок со счётчиками + поиск. Карточка пробела:
название, артикулы, бейдж состояния, сводка доступных источников. Действия с
явной подписью судьбы исходника у каждой кнопки:
«Перенести из реальных» (для Personal/Defect — «уйдёт из Реальных»; для
Эталонных — «останется»), «Из свободных» («останется»), «Апгрейдить»,
«Поиск/вручную». Пробел показывается всегда; если источников нет — единственное
действие «Поиск/вручную», без пустых кнопок.
